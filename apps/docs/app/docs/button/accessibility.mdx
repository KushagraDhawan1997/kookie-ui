# Accessibility

The Button component is built with comprehensive accessibility features that work for everyone, regardless of ability or interaction method. It implements systematic color contrast, semantic markup, keyboard navigation, and assistive technology support out of the box.

## Contrast

The Button component uses a systematic color approach that ensures all button combinations maintain minimum 3:1 contrast ratios. However, different variants provide different levels of accessibility compliance depending on your specific needs.

**Standard buttons** maintain 3:1 minimum ratios but do not meet WCAG 2.1 AA standards for normal text. Most variants meet AA standards for large text, making them suitable for headings and prominent UI elements. For general interface text requiring AA compliance, consider using the `highContrast` prop to automatically enhance contrast ratios.

**Solid buttons** provide the highest contrast among standard variants, using white text on colored backgrounds for maximum visibility. **All other variants** use carefully chosen text colors that maintain proper contrast against their respective backgrounds.

<Callout.Root size="2" variant="soft" color="gray" my="8">
  <Callout.Text>Buttons with transparent backgrounds require evaluation against their specific surrounding content to determine true contrast compliance.</Callout.Text>
</Callout.Root>

### High Contrast

The `highContrast` prop automatically ensures maximum visibility for users with visual impairments or challenging viewing conditions. When enabled, it increases color saturation, improves contrast ratios, and enhances focus indicators across all variants.

**High-contrast variants with non-transparent backgrounds** (Classic, Solid, Soft, and Surface) demonstrate excellent contrast ratios and meet both [WCAG 2.1 AA](https://www.w3.org/WAI/WCAG21/quickref/?versions=2.1&levels=aa) and [AAA](https://www.w3.org/WAI/WCAG21/quickref/?versions=2.1&levels=aaa) standards for normal and large text, providing comprehensive accessibility compliance.

**High-contrast buttons with transparent backgrounds** (Outline and Ghost) still require evaluation against the specific background color they are placed upon to determine their true contrast ratio and WCAG compliance.

## Keyboard Navigation

The Button component automatically supports full keyboard accessibility with standard interaction patterns. **Space key** activates buttons. **Tab navigation** moves between buttons in logical order. **Focus indicators** automatically use 2px outlines with appropriate colors and offsets per variant.

The focus-visible implementation automatically shows focus rings only for keyboard navigation, not mouse clicks. This provides clear navigation cues without visual clutter.

      <Button size="3" variant="ghost">Button</Button>

## Screen Reader

The Button component automatically uses semantic `<button>` elements with proper ARIA attributes for comprehensive screen reader compatibility. All button states automatically communicate clearly to assistive technology.

**Loading states** automatically use `aria-busy="true"`, `aria-disabled="true"`, `aria-describedby`, and dynamic `aria-label` to announce processing status. The component intelligently extracts button text and creates comprehensive loading announcements while a visual spinner indicates progress.

<CodeBlock

  preview={
    <Button size="3" variant="solid" loading>Processing Order...</Button>
  }
>
</CodeBlock>

**Disabled states** automatically use proper `disabled` attributes and visual indicators. Screen readers automatically announce the unavailable state while visual styling prevents interaction confusion.

<CodeBlock

  preview={

      <Button size="3" variant="solid" disabled>Disabled</Button>
      <Button size="3" variant="outline" disabled>Disabled</Button>
    </Flex>

## Touch Targets

The Button component automatically accommodates users with varying motor abilities. All sizes automatically meet or exceed accessibility guidelines for minimum touch targets.

**Size Guidelines:**
- **Size 1 (24px):** Suitable for dense desktop interfaces only
- **Size 2 (32px):** Suitable for standard desktop interfaces  
- **Size 3 (40px):** Recommended for mobile interfaces and important actions
- **Size 4 (48px):** Recommended for hero sections and maximum accessibility

<Callout.Root size="2" variant="soft" color="gray" my="8">
  <Callout.Text>Mobile interfaces should avoid sizes 1 and 2. They do not meet the minimum touch target guidelines.</Callout.Text>

## Loading

The loading state automatically provides comprehensive accessibility support beyond basic visual feedback.

<CodeBlock

  preview={
    <Button size="3" loading>Processing...</Button>
  }
>
</CodeBlock>

**Automatic accessibility features:**
- `aria-busy="true"` automatically announces processing state
- `aria-disabled="true"` automatically prevents interaction
- `aria-describedby` automatically links to loading announcement
- `aria-label` dynamically updates to include "(loading)" status
- VisuallyHidden text automatically includes: "[Button text] is loading, please wait..."
- Original button text automatically remains available to screen readers
- Intelligent text extraction works with both string and React element children

</Section>

## Text Extraction

The Button component automatically extracts meaningful text for screen readers from both string children and React elements. If no text can be extracted, it falls back to "button" to ensure screen readers always have content to announce.
